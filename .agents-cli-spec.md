# App / Agent Specification: agent-dev-team

## Summary
The "agent-dev-team" is an ADK multi-agent software-development system (an AI dev team) built with Google ADK in Python. It takes an application's requirements as input and produces working, tested, and deployed code. Its first target job is to build a "Study Tracker" web app.

## Motivation
Building software traditionally requires human developers, PMs, testers, and QA engineers working iteratively. This process is time-consuming and expensive. The "agent-dev-team" aims to automate this software development lifecycle by using specialized AI agents to handle product management, test writing, coding, specification review, and end-to-end quality assurance. This reduces development time and ensures a systematic approach to building and deploying apps.

## Guide-level explanation (the user experience)
The user provides the "agent-dev-team" with a set of requirements (like the Study Tracker appendix) via the Orchestrator agent.
First, the PM agent takes the requirements and generates a structured specification containing features, data contracts, acceptance criteria, and the tech stack.
Next, the Test-writer agent creates executable unit and integration tests based on the spec.
Then, the Coder agent writes the application code to pass those tests.
After coding, tests are run. The Spec-Reviewer agent also statically checks the code against the spec and production pitfalls. If tests fail or the review finds violations, the system loops back to the Coder agent to fix issues (up to a maximum of 5 iterations).
Once all tests and reviews pass (green state), the E2E-QA agent runs Stage 1 tests against the locally running app by automating a real browser to click through the interface. If this fails, it loops back to the Coder.
Once Stage 1 E2E passes, the app is deployed to the target hosting platform. The E2E-QA agent then runs Stage 2 tests against the deployed production app. If successful, the user receives the final deployed app.

## Reference-level explanation (technical architecture)
- **App type:** agent / multi-agent system (event-driven pipeline)
- **Frontend:** none
- **Backend / framework:** Google ADK, Python
- **Database:** none
- **Authentication:** none
- **Hosting / deployment:** Runs LOCALLY to generate the target app. (Only the target app gets deployed to GCP).
- **API style:** none
- **LLM provider/family:** Google Gemini family — verify exact current model at build time via the agents-cli model listing.
- **Secrets management:** Google Secret Manager in any deployed context; local dev uses an uncommitted `.env`.

The system operates via a fixed pipeline with a test-fix loop, driven by the orchestrator (NOT dynamic LLM routing). 
Pipeline topology: PM -> Test-writer -> Coder -> run tests -> Spec-Reviewer -> (if fail/violation: loop back to Coder, max 5 iterations) -> (when green: E2E-QA Stage 1 local) -> (if fail: loop back to Coder) -> Deploy -> (E2E-QA Stage 2 production) -> accept.

The system consists of 6 independent ADK agents:
1. Orchestrator (root agent): runs pipeline, holds shared session state, controls fix-loop, decides when done.
2. PM agent: converts requirements to structured spec.
3. Test-writer agent: writes executable unit/integration tests (the only one that writes test files).
4. Coder agent: writes/edits app code (never edits test files).
5. Spec-Reviewer agent (static): checks coder output vs spec + deploy "Production Pitfalls" lessons. Rejects loudly on data-contract mismatch.
6. E2E-QA agent: uses Playwright MCP to drive a real browser for Stage 1 (local) and Stage 2 (production) visual and interaction testing.

## Data Contracts & Pipeline I/O
- **Boundary: User -> Orchestrator**
  - **Input schema:** `requirements` (string/markdown)
  - **Output schema:** Final application code and deployment
  - **Example Input:** The Study Tracker appendix markdown text.
  - **Example Output:** Deployed Study Tracker application.

- **Boundary: Orchestrator -> PM**
  - **Input schema:** `requirements` (string/markdown)
  - **Output schema:** `Spec` = `{ features: [{id:str, title:str, description:str}], dataContracts: [{node:str, input:object, output:object, example:object}], acceptanceCriteria: [{id:str, given:str, when:str, then:str}], techStack: {frontend:str, backend:str, db:str, auth:str, hosting:str} }`
  - **Example Input:** "Build a Study Tracker web app with today, week, month views..."
  - **Example Output:** `{ "features": [{"id":"f1", "title":"Today View", "description":"One-tap toggle for study/rest"}], "dataContracts": [{"node": "Session", "input": {"type": "study"}, "output": {"id": "uuid"}, "example": {"id": "123", "type": "study"}}], "acceptanceCriteria": [{"id": "a1", "given": "On Today view", "when": "Clicks toggle", "then": "Switches state"}], "techStack": {"frontend":"Next.js", "backend":"FastAPI", "db":"Cloud SQL (PostgreSQL)", "auth":"Firebase Auth", "hosting":"Cloud Run"} }`

- **Boundary: PM -> Test-writer**
  - **Input schema:** `Spec` (JSON object)
  - **Output schema:** `TestSuite` = `{ framework: str, files: [{path:str, content:str}], coversCriteria: [str] }`
  - **Example Input:** The `Spec` JSON from the PM agent.
  - **Example Output:** `{ "framework": "pytest", "files": [{"path": "backend/tests/test_sessions.py", "content": "def test_record_session(): pass"}], "coversCriteria": ["a1"] }`

- **Boundary: (Spec + TestSuite + TestResults) -> Coder**
  - **Input schema:** `Spec` (JSON), `TestSuite` (JSON), `TestResults` (JSON)
  - **Output schema:** `CodeChange` = `{ summary: str, files: [{path:str, content:str, action:"create"|"modify"|"delete"}] }`
  - **Example Input:** Spec JSON, TestSuite JSON, and a TestResults JSON indicating a failed test due to missing endpoint.
  - **Example Output:** `{ "summary": "Implement session endpoint", "files": [{"path": "backend/main.py", "content": "...", "action": "modify"}] }`

- **Boundary: Test-runner (tool) -> state**
  - **Input schema:** Execution command for tests
  - **Output schema:** `TestResults` = `{ passed: int, failed: int, failures: [{test:str, message:str, trace:str}], logs:str }`
  - **Example Input:** `pytest backend/tests`
  - **Example Output:** `{ "passed": 0, "failed": 1, "failures": [{"test": "test_record_session", "message": "404 Not Found", "trace": "..."}], "logs": "..." }`

- **Boundary: (Spec + CodeChange) -> Spec-Reviewer**
  - **Input schema:** `Spec` (JSON), `CodeChange` (JSON)
  - **Output schema:** `Review` = `{ compliant: bool, violations: [{rule:str, file:str, detail:str, severity:"block"|"warn"}] }`
  - **Example Input:** Spec JSON, CodeChange JSON showing new endpoint implementation.
  - **Example Output:** `{ "compliant": false, "violations": [{"rule": "data contract mismatch", "file": "backend/main.py", "detail": "Missing timezone field in Session object", "severity": "block"}] }`

- **Boundary: (appUrl + acceptanceCriteria) -> E2E-QA**
  - **Input schema:** `appUrl` (str), `acceptanceCriteria` (List of JSON objects)
  - **Output schema:** `E2EReport` = `{ stage:"local"|"production", passed:int, failed:int, failures:[{criterion:str, screenshotPath:str, error:str}] }`
  - **Example Input:** `http://localhost:3000`, `[{"id":"ac1", "given":"User on Today tab", "when":"clicks toggle", "then":"segment recorded"}]`
  - **Example Output:** `{ "stage": "local", "passed": 1, "failed": 0, "failures": [] }`

## Tools & Integrations
- **Google ADK runtime + Gemini API**
  - **Purpose:** Core agent intelligence and execution framework.
  - **Endpoint / SDK:** Google ADK Python SDK, Gemini API.
  - **Authentication:** Service account / API key in env, via Google Secret Manager in any deployed context; local dev uses an uncommitted `.env`.
- **Playwright MCP server**
  - **Purpose:** E2E-QA browser automation for testing the app locally and in production.
  - **Endpoint / SDK:** MCP Protocol.
  - **Authentication:** None (no external auth required).
- **Test-runner tool**
  - **Purpose:** Executing unit/integration tests written by the Test-writer agent.
  - **Endpoint / SDK:** CLI (pytest for backend, JS test runner for frontend).
  - **Authentication:** None.

## Constraints & Safety Rules
- The Coder agent MUST NEVER edit or create files under the test directory; only the Test-writer agent writes tests.
- The Spec-Reviewer agent MUST reject loudly on data-contract mismatch; it must never silently coerce to defaults.
- Fix-loop hard cap: The system allows a maximum of 5 iterations for the fix-loop. Once reached, it must stop and report to the human to avoid infinite loops and cost runaway.
- There is an overall LLM cost/budget ceiling per run; the system must stop and report if exceeded.
- E2E-QA Stage 2 must use a DEDICATED test account on production; it must never perform destructive actions on real users' data.
- Secrets must never be hard-coded or committed; they are loaded from the environment / Google Secret Manager as centralized config.

## Success Criteria
- The pipeline produces a Study Tracker where 100% of the acceptance criteria pass in Stage 1 local E2E.
- Deployment to GCP succeeds and Stage 2 production E2E passes 100% of acceptance criteria.
- Pipeline reaches a green state within the 5-iteration cap on the Study Tracker job.
- No data-contract mismatch escapes the Spec-Reviewer (every mismatch is flagged, none are coerced to defaults).

## Drawbacks
- A fixed pipeline orchestration topology is less flexible than dynamic LLM routing, meaning the system may struggle with edge cases that do not cleanly fit the PM -> Test -> Code -> Review sequence.
- The 5-iteration cap might lead to failure on complex tasks that require more steps to resolve.
- Maintaining test separation (Coder vs Test-writer) can sometimes lead to misalignment if the Test-writer writes poorly conceived tests, forcing the Coder to chase impossible constraints until the cap is hit.

## Rationale & alternatives
The orchestrator-driven fixed pipeline with a test-fix loop was chosen over dynamic LLM routing because it provides predictability, easier debugging, and enforces strict separation of concerns (e.g., Coder vs. Test-writer). This systematic approach mirrors human software development processes and prevents agents from getting lost in unbounded dynamic planning loops. Google ADK was selected as the framework because it provides robust primitives for building this type of pipeline.

## Prior art
Existing coding assistants like GitHub Copilot or Cursor offer autocomplete and localized code generation, but they are not autonomous multi-agent systems that handle the full lifecycle from requirements to deployment. Other agentic coding systems exist, but the "agent-dev-team" differentiates itself by strictly enforcing a test-driven development pipeline, static specification review, and comprehensive E2E visual QA via Playwright, ensuring a higher quality, deployment-ready output.

## Unresolved questions

None — every product and technical decision was resolved with the user during the discovery interview before this spec was written. (Open day-boundary nuance was decided: day = user's local-timezone midnight.)

## Future possibilities
- Version 2.0 could support dynamic sub-task routing for the Coder agent (e.g., splitting frontend and backend work into parallel agent tasks).
- Expanding the Spec-Reviewer to include security auditing and performance profiling.
- Supporting a wider range of target app tech stacks by dynamically generating the test-runner and deployment scaffolding.

## Appendix: Layer 2 - Study Tracker Target App
*(This appendix serves as the concrete input `requirements` that the agent-dev-team will consume to build the app)*

**Background:** There is an existing, published React artifact prototype (single-user, browser-storage only, no auth, no backend): https://claude.ai/public/artifacts/29977f06-dcee-473d-925c-d1c1cbe3fa69 . Its feature inventory and known limitations are documented in the knowledge base at `notes/2026-06-14-study-tracker-app-summary.md` and `notes/2026-06-14-artifact-architecture-limitations.md`. The agent-dev-team must rebuild it as a real multi-user web app. A reference weekly-timetable image (provided by the user) defines the target look of the new Week calendar heatmap.

### Features to Keep
- **Today tab:** ONE-TAP big toggle button switches between "studying" and "rest"; every switch pops a modal to write a note ("what did you study?" / "what was the break for?", skippable).
- **Three-tab structure:** The Today / Week / Month layout.
- **Month:** "last 30 days" summary + activity heatmap.

### Defects to Fix
- **Today:** Add ability to manually edit / backfill / adjust a study or rest segment (user sometimes forgets to stop the timer). Currently, there is NO way to customize/adjust a recorded segment.
- **Today:** The min (4h) and ideal (6h) goals are currently hardcoded; make them user-editable settings.
- **Week:** Replace the bar chart with a CALENDAR HEATMAP (columns = days of the week, rows = time slots of the day, each colored block = what a given time segment was used for, similar to a weekly timetable). Clicking a block shows what it was + how many hours/minutes. Add navigation to previous/next week.
- **Month:** Add previous/next month navigation (currently cannot switch months).
- **Timezone Bug:** Today uses the system local time, but the Month map used UTC, causing segments after a certain local hour to roll into the wrong day. FIX: bucket every record by day using the USER'S LOCAL TIMEZONE midnight (00:00) consistently across Today/Week/Month. Day boundary = local midnight.

### New Architecture Features
- **Authentication:** Firebase Auth supporting Google account / email / phone signup+login. Purpose: sync the same user's data across phone and computer.
- **Backend + persistence:** A real backend so data is stored server-side.
- **Multi-user, deployable, shareable:** Deploy to Google Cloud so the user can share it and friends can register their own accounts and track their own time.

### Tech Stack
- **Frontend:** Next.js (React) — reuse the existing React component logic.
- **Backend:** Python FastAPI (REST API).
- **Database:** Cloud SQL (PostgreSQL) — relational; good for weekly/monthly aggregation queries.
- **Authentication:** Firebase Auth (Google / email / phone); FastAPI verifies the Firebase-issued token.
- **Hosting:** Google Cloud Run (containerized).
- **API style:** REST.
- **Secrets management:** Google Secret Manager for deployed envs; local dev uses an uncommitted `.env`.

### Data Schema / Contract
- `Session`: `{ id: uuid, userId: str, date: "YYYY-MM-DD" (local-tz midnight bucketed), start: ISO-timestamp(with tz), end: ISO-timestamp(with tz), type: "study" | "rest", notes: str|null }`
- `UserSettings`: `{ userId: str, minHours: number, idealHours: number, timezone: str(IANA) }`
- **Constraint:** Every server query MUST be scoped to the authenticated `userId` (server-side multi-user isolation).

### Success Criteria (Study Tracker)
1. One tap correctly records study vs rest plus what the time was used for.
2. Today / Week / Month all support the interactions described above (editable goals, backfill, navigable week calendar with clickable blocks, navigable month, correct local-tz day bucketing).
3. A backend persists the data per the schema.
4. After full deployment to GCP, a friend can register an account and correctly record their own time.
